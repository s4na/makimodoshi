# makimodoshi — 要件定義書

## Context

Railsアプリ開発中、ブランチを切り替えると「切り替え前のブランチで実行したマイグレーションがDBに残っている」状態になることがある。この状態で `rails s` を起動すると、schema.rbとDBの不整合が原因で予期しない挙動やエラーが発生する。

現状、開発者はこの不整合に気づいたら手動で `db:rollback` を実行するか、`db:reset` でやり直す必要がある。これは面倒であり、不整合に気づかないまま開発を続けてしまうリスクもある。

さらに、マイグレーションファイルが手元に存在しない（ブランチ切り替えで消えた）場合や、誤って編集してしまった場合は、通常の `db:rollback` では正しく戻せない。

## Goal

- `rails s` 起動時に、DBのマイグレーション状態がschema.rbより先行している場合、自動的にロールバックして整合性を保つ
- マイグレーションファイルが存在しない・変更されていても、保管済みの情報でロールバックできる
- 開発者が任意のタイミングで手動ロールバックできるコマンドを提供する

## In Scope

- `rails s` 起動時の自動ロールバック（development環境のみ）
- マイグレーション実行時（`db:migrate`）に、ロールバックに必要な情報を隠しテーブルに自動保管
- 隠しテーブルは `schema.rb` のダンプ対象から除外する
- 手動ロールバックコマンド（1つ戻す / マイグレーションIDを指定して戻す）
- 保管済みマイグレーション一覧の確認コマンド
- development以外の環境でインストールされた場合の警告表示
- `schema.rb` を使用するプロジェクトへの対応

## Out of Scope

- `structure.sql` を使用するプロジェクトへの対応
- マイグレーションの順方向実行（migrate）の代替
- production / staging 環境での動作
- DBが schema.rb より古い場合の対応（マイグレーション不足の検知）
- 確認プロンプト（自動ロールバック時のY/n確認）

## User Stories

1. **ブランチ切り替え後の自動修復**: 開発者として、feature-Aブランチからmainブランチに切り替えた後に `rails s` を実行したとき、feature-Aで追加されたマイグレーションが自動的にロールバックされてほしい。手動で `db:rollback` を繰り返す手間をなくしたい。

2. **ファイル不在でもロールバック**: 開発者として、マイグレーションファイルがブランチ切り替えにより手元に存在しなくなっても、保管された情報を使って正しくロールバックできてほしい。

3. **誤編集からの復旧**: 開発者として、マイグレーションファイルを誤って編集してしまった場合でも、保管された元の情報を使って1つロールバックし、やり直したい。

4. **手動ロールバック**: 開発者として、`rails s` の起動時以外にも、任意のタイミングでrailsコマンドから1つ戻したり、特定のマイグレーションIDを指定してロールバックしたい。

5. **マイグレーション状態の確認**: 開発者として、隠しテーブルに保管されているマイグレーション情報の一覧を確認して、どのマイグレーションがロールバック可能かを把握したい。

6. **安全な環境制限**: 開発者として、このgemがdevelopment以外の環境で誤って動作しないことを保証してほしい。development以外でインストールされた場合は警告を受け取りたい。

## Acceptance Criteria

### 主要シナリオ

```gherkin
Feature: rails s 起動時の自動ロールバック

  Scenario: DBがschema.rbより先行している場合に自動ロールバックされる
    Given development環境でRailsアプリが存在する
    And schema.rbのversionが "20240101000000" である
    And DBのschema_migrationsに "20240101000000" と "20240201000000" が存在する
    And "20240201000000" のロールバック情報が隠しテーブルに保管されている
    When `rails s` を実行する
    Then "20240201000000" が自動的にロールバックされる
    And 標準出力にロールバック実行のメッセージが表示される
    And Railsサーバーが正常に起動する

  Scenario: DBとschema.rbが一致している場合は何もしない
    Given development環境でRailsアプリが存在する
    And schema.rbのversionとDBのschema_migrationsの最大値が一致している
    When `rails s` を実行する
    Then ロールバックは実行されない
    And 追加の出力は表示されない
    And Railsサーバーが正常に起動する

  Scenario: db:migrate実行時にロールバック情報が保管される
    Given development環境でRailsアプリが存在する
    And 新しいマイグレーションファイルが存在する
    When `rails db:migrate` を実行する
    Then マイグレーションが正常に実行される
    And ロールバックに必要な情報が隠しテーブルに保管される
```

### 例外シナリオ

```gherkin
  Scenario: マイグレーションファイルが存在しなくてもロールバックできる
    Given development環境でRailsアプリが存在する
    And DBに "20240201000000" のマイグレーションが適用されている
    And "20240201000000" のロールバック情報が隠しテーブルに保管されている
    And "20240201000000" のマイグレーションファイルが存在しない
    When 自動ロールバックまたは手動ロールバックが実行される
    Then 隠しテーブルの情報を使って正しくロールバックされる

  Scenario: development以外の環境では動作しない
    Given production環境でRailsアプリが存在する
    And このgemがGemfileに含まれている
    When Railsが起動する
    Then 自動ロールバックは実行されない
    And 警告メッセージが表示される
```

## NFR（非機能要件）

| 区分 | 要件 |
|---|---|
| **性能** | DBとschema.rbが一致している場合（通常ケース）、起動時間への影響を最小限にする（schema.rbの読み取り1回 + DBクエリ1回程度） |
| **性能** | ロールバック実行時の所要時間はマイグレーション内容に依存するため、gem側での制約は設けない |
| **安全性** | development環境以外では一切のDB操作を行わない |
| **安全性** | 隠しテーブルは `schema.rb` のダンプ対象から除外し、ホストアプリのスキーマを汚染しない |
| **互換性** | Rails 6.1以上、Ruby 3.0以上をサポート |
| **運用** | ログ出力は `puts` で標準出力に行う |

2. 隠しテーブルのデータが存在しないマイグレーション（gem導入前に実行されたもの）のロールバックが要求された場合、どう振る舞うべきか？（スキップ？エラー？）
3. 手動ロールバックコマンドの具体的なコマンド名・引数形式はどうするか？（例: `rails makimodoshi:rollback`、`rails makimodoshi:rollback VERSION=xxx`、`rails makimodoshi:status`）
